### TDD 流程（如果 `is_rework` 为 FALSE — 新 feature）

1. **阅读**现有代码，理解项目规范（命名、错误处理、代码风格）
2. **先写测试（RED）**：
   测试必须从 feature 的**验收标准**派生，而不是从代码结构派生。

   **必须测试**（业务价值）：
   - 业务规则和领域逻辑（计算、状态转换、有业务含义的校验）
   - 边界情况和错误场景（非法输入、边界条件、并发访问）
   - API 契约（请求/响应格式、状态码、错误响应）
   - 包含业务逻辑的数据转换

   **不要测试**（零价值）：
   - 构造函数、getter、setter、简单访问器
   - 没有业务逻辑的简单 CRUD（仅委托给数据层）
   - 框架行为（如依赖注入是否工作、ORM 是否提交）
   - 仅声明 bean/配置的配置类

   **判断标准**：「如果这个测试失败，会指示什么业务问题？」— 没有清晰答案就跳过。

3. **运行测试 — 确认 FAIL**：`{{test_command}}` — 测试应该失败（还没实现）
4. **实现（GREEN）**，遵循该技术栈的最佳实践
5. **运行测试 — 确认 PASS**：`{{test_command}}` — 所有测试应该通过
6. **更新** {{workspace}}/task_list.json：将 `passes` 设为 `true`
7. **提交**：`git add -A && git commit -m "<feature-id>: <简短描述>"`
